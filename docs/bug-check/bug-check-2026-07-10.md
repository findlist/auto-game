## 三、发现的问题列表（按严重程度排序）

### 🔴 严重问题（16 个）

#### S1. Service Worker 缓存版本与应用版本不一致
- **文件**：[sw.js](file:///e:/work/auto-game/public/sw.js#L2) 第 2 行
- **问题描述**：`CACHE_VERSION = 'color-sort-v1.13.0'`，但 `package.json` version 为 `1.12.0`，`AboutPage.tsx` 和 `SettingsPage.tsx` 显示也是 `v1.12.0`。三个版本号互相矛盾（sw.js 1.13.0 / package.json 1.12.0 / 导出存档 1.1.0）。
- **影响**：无法通过缓存版本判断线上应用版本；用户升级时可能复用旧缓存看到旧版页面；维护时难以追踪缓存与代码对应关系。
- **修复建议**：建立版本同步机制，构建时从 `package.json` 动态注入 version 到 sw.js，或通过脚本保证三者一致。

#### S2. Service Worker 预缓存 `addAll` 原子性失败导致离线功能失效
- **文件**：[sw.js](file:///e:/work/auto-game/public/sw.js#L20-L22) 第 20-22 行
- **问题描述**：`cache.addAll(STATIC_ASSETS).catch(() => {})` —— `Cache.addAll()` 是原子操作，若预缓存列表中任一资源（如 `/og-image.png`、`/offline.html` 等）不存在或请求失败，**整个 addAll 操作 reject**，所有预缓存资源均未写入。`.catch(() => {})` 静默吞错，SW 照常激活但离线缓存为空。
- **影响**：任一预缓存资源缺失时，首次离线访问 `caches.match` 全部返回 undefined，回退到 `/offline.html` 也因未缓存而失败，用户离线看到浏览器默认离线页。
- **修复建议**：改为 `Promise.allSettled(STATIC_ASSETS.map(url => cache.add(url)))`，单个资源失败不影响其他资源预缓存。

#### S3. visitTracker 会话时长重复累加，统计数据严重失真
- **文件**：[visitTracker.ts](file:///e:/work/auto-game/src/game/visitTracker.ts#L90-L104) 第 90-104 行；调用方 [main.tsx](file:///e:/work/auto-game/src/main.tsx#L26-L31) 第 26-31 行
- **问题描述**：`recordSessionDuration` 每次调用执行 `data.totalSessionDuration += duration`（duration = Date.now() - lastSessionStart），但调用后**未重置 lastSessionStart**。main.tsx 中该函数同时绑定 `beforeunload` 和 `visibilitychange(hidden)`：
  - 用户切换标签页 → 累加从会话开始到现在的全部时长
  - 切回再切走 → **从会话开始到现在的全部时长被再次累加**
  - 页面关闭时 `beforeunload` 和 `visibilitychange` 可能同时触发，一次关闭累加两次
- **影响**：`totalSessionDuration` 严重虚高，`getAvgSessionDuration()` 返回值远超实际。例如 10 分钟会话，切换标签 3 次后累计可能变成 40+ 分钟。StatsPage 显示的平均会话时长完全不可信。
- **修复建议**：累加后更新 `lastSessionStart = Date.now()`（下次只计算增量），或设置标志位防止同一会话重复记录。

#### S4. GameBoard 通关动画多个 setTimeout 未清理，组件卸载后仍执行
- **文件**：[GameBoard.tsx](file:///e:/work/auto-game/src/components/GameBoard.tsx) 第 224、227、275-278、280、305、308、309 行
- **问题描述**：`handleTubeClick` 内多处直接调用 `setTimeout(...)` 不保存定时器 ID，组件卸载时不清理。包括：
  - `setTimeout(() => setPouringTo(null), 300)`
  - `setTimeout(() => setMovesPulse(false), 300)`
  - `newTubes.forEach((_, idx) => setTimeout(..., idx * 80))`（逐个标记 settled）
  - `setTimeout(() => setSettledTubes(new Set()), 2000)`
  - `setTimeout(() => SoundEngine.star(), 600 + i * 200)`
  - `setTimeout(() => setShowParticles(false), 3000)`
  - `setTimeout(() => onWin(...), 500)`
- **影响**：通关动画期间返回首页（组件卸载），定时器仍触发 setState 到已卸载组件，甚至调用已失效的 onWin 回调；进入新关卡时旧关卡的 setSettledTubes 会干扰新关卡显示。
- **修复建议**：用 `useRef` 收集所有定时器 ID，在 `useEffect` cleanup 中统一清理；或把通关动画逻辑抽到独立 `useEffect`，由 `isWon` 触发。

#### S5. 回放自动播放定时器 replayTimerRef 组件卸载时未清理
- **文件**：[GameBoard.tsx](file:///e:/work/auto-game/src/components/GameBoard.tsx#L65) 第 65 行（定义）、第 734 行（设置）
- **问题描述**：`replayTimerRef.current = setTimeout(stepNext, 500)` 在自动播放循环中不断调度，但 GameBoard 没有在卸载时清理它的 useEffect。用户若通过"返回首页"直接离开，定时器不会被清除。
- **影响**：组件卸载后定时器继续触发 `setReplayTubes`、`setReplayStep`，且 `stepNext` 内部会继续 `setTimeout(stepNext)` 形成无法终止的回调链。
- **修复建议**：增加组件级 `useEffect`，cleanup 时 `if (replayTimerRef.current) clearTimeout(replayTimerRef.current)`。

#### S6. 回放"自动播放"按钮文本不随播放状态更新

#### S7. 限时模式通关后自动跳关，胜利弹窗一闪而过
- **文件**：[App.tsx](file:///e:/work/auto-game/src/App.tsx#L467-L482) 第 467-482 行
- **问题描述**：`handleWin` 内对限时模式直接调用 `setTimedScore(newScore)`，而 `timedScore` 是 GameBoard 的 prop 且在其 `useEffect` 依赖数组中。timedScore 变化触发 GameBoard 重新生成关卡（执行 setIsWon(false) 等），导致胜利弹窗 500ms 后消失。`handleNextLevelAction` 中限时模式分支（第 596-601 行）的 `setTimedScore(s => s + 1)` 永远不会执行，成为死代码。
- **影响**：限时模式通关后玩家来不及看到胜利信息和点击按钮就被强制进入下一关。
- **修复建议**：从 `handleWin` 中移除 `setTimedScore(newScore)`，只更新最高分，让 timedScore 累加交给用户点击"下一关"时的 `handleNextLevelAction`。

#### S8. tubesRef.current = tubes 在渲染期间直接修改 ref
- **文件**：[GameBoard.tsx](file:///e:/work/auto-game/src/components/GameBoard.tsx#L82) 第 82 行
- **问题描述**：`tubesRef.current = tubes;` 写在函数组件渲染体内，每次渲染执行。React 18 严格模式下渲染执行两次，并发模式下渲染可能被中断重试，导致 ref 与实际 tubes 短暂不一致。
- **影响**：App 的 `handleHint` 通过 `currentTubesRef.current` 读取当前 tubes，在严格模式下可能读到中间态值。
- **修复建议**：改为在 `useEffect` 中赋值：`useEffect(() => { tubesRef.current = tubes; }, [tubes]);`

#### S9. 关卡生成器不可解兜底仍返回不可解关卡
- **文件**：[levelGenerator.ts](file:///e:/work/auto-game/src/game/levelGenerator.ts#L100-L125) 第 100-125 行（generateLevel）；同样模式见 [levelGenerator.ts](file:///e:/work/auto-game/src/game/levelGenerator.ts#L192-L216) 第 192-216 行（generateEndlessLevel）、[levelGenerator.ts](file:///e:/work/auto-game/src/game/levelGenerator.ts#L274-L298) 第 274-298 行（generateTimedLevel）、[dailyChallenge.ts](file:///e:/work/auto-game/src/game/dailyChallenge.ts#L76-L101) 第 76-101 行

#### S10. 每日排行榜截断逻辑丢失历史最佳成绩

#### S11. 每日提示道具领取非幂等，可重复领取

#### S12. 关卡编辑器颜色整除校验逻辑错误
- **文件**：[levelEditor.ts](file:///e:/work/auto-game/src/game/levelEditor.ts#L127-L133) 第 127-133 行
- **问题描述**：`validateLevel` 注释说"每种颜色层数必须可整除填满试管"，但实际逻辑 `tubes.some(t => t.capacity === count)` 判断的是"颜色层数是否等于任意一个试管容量"，而非整除。当一种颜色层数是容量的整数倍但大于单个试管容量时（如容量 4、颜色 A 有 8 层填满 2 个试管），校验误判为无效。
- **影响**：玩家创建的合法关卡被误判无效无法保存；校验消息误导。

#### S13. 回放视频生成无错误处理 + ObjectURL 内存泄漏

#### S14. 空步数回放无法解码

#### S16. solver BFS 存储完整路径导致内存浪费和性能问题
- **文件**：[solver.ts](file:///e:/work/auto-game/src/game/solver.ts#L76-L99) 第 76-99 行
- **问题描述**：`solveBFS` 队列存储 `{ tubes: Tube[][], steps }`，其中 `Tube[][]` 是完整路径（`[...path, nextTubes]`）。但路径仅用于 `current = path[path.length - 1]`，完整路径从未使用。每次扩展都 `[...path, nextTubes]` 拷贝整个历史，内存随深度指数增长。
- **影响**：复杂关卡搜索时内存占用极高，可能导致移动端 OOM 或搜索提前触及 maxNodes 限制，影响可解性判断准确性。
- **修复建议**：队列只存储当前状态 `{ tubes: Tube[], steps }`（去掉路径），或用父节点指针重建路径。
